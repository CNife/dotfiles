# 交互式创建待办

在今天的 Obsidian 工作日志中，通过对话引导用户创建结构化的待办事项。

## 流程

### 第一步：定位日志文件

执行以下脚本获取今天的日志路径：

```python
python3 -c "
import os, shutil
from datetime import datetime
weekdays = ['星期一', '星期二', '星期三', '星期四', '星期五', '星期六', '星期日']
now = datetime.now()
log_file = f'/mnt/c/Obsidian/工作/工作日志/{now.year}/{now.month:02d}/{now.year} 年 {now.month} 月 {now.day} 日 {weekdays[now.weekday()]}.md'
if not os.path.exists(log_file):
    os.makedirs(os.path.dirname(log_file), exist_ok=True)
    shutil.copy('/mnt/c/Obsidian/工作/工作日志/日记.md', log_file)
print(f'LOG_FILE={log_file}')
"
```

脚本会自动创建日志文件（若不存在）。接下来读取 `LOG_FILE` 内容，了解已有待办和项目上下文。

### 第二步：搜索背景（关键步骤）

**不要空手问用户问题。** 用户提到项目或主题后，先搜索相关背景：

1. 在工作日志目录中搜索项目关键词，了解近期进展
2. 读取最近 3-5 天的日志，理解用户正在做什么
3. 搜索笔记和技术方案目录中是否有相关文档

搜索完成后，带着理解去和用户对话。

### 第三步：交互式追问

通过简洁的追问明确待办内容。原则：

- **选择题优于开放题**：给出 2-4 个选项让用户选
- **一次只问一个关键问题**：不要列一堆问题
- **已经能从日志中推断的信息不要问**
- **聚焦"用户需要做什么"**，不要把背景信息、其他人的事当成待办

### 第四步：生成待办并写入

按照以下格式写入日志的 `# 待办事项` 区块之后（如果已有待办，追加到列表末尾）。

**格式规范：**

```markdown
# 项目名

## 任务标题

背景说明（简短，不是待办项）

- [ ] 具体待办 1
- [ ] 具体待办 2
```

**层级规则：**
- 一级标题 `#` → 项目名（如 `# Genos Reg Server`）
- 二级标题 `##` → 任务标题（如 `## 正式环境部署`）
- 背景说明写在二级标题下面，不是待办项
- `- [ ]` 列表写用户自己要做的事

**内容规则：**

- ✅ 每条待办是一个具体动作，不笼统
- ✅ 只写用户要做的事，不写别人的事（别人的事放背景说明里）
- ✅ 粒度适中：用户说要简化就简化，说太粗就细化
- ❌ 不要写成实施手册（不要"安装 Python 3.10"、"pip install xxx"）
- ❌ 不要包含等待/阻塞项作为待办（放背景说明）

**示例：**

好的写法：
```
## 正式环境部署

吴雯在申请新机器，机器到位后需要部署后端。

- [ ] 安装 GPU 驱动，配置 conda 环境，部署后端
- [ ] 拷贝依赖数据文件
- [ ] 修改前端配置，连接新后端
- [ ] 联调验证，正式上线
```

应避免的写法：
```
- [ ] 等吴雯申请新机器到位          ← 别人的事，不是待办
- [ ] 安装 Python 3.10              ← 太细，不是关键结果
- [ ] pip install fastapi uvicorn   ← 实施手册式写法
- [ ] 完成部署                      ← 太笼统
```

### 第五步：迭代修正

用户可能对生成的待办有修改意见（太细、太粗、格式不对等）。直接修改，不要反复确认。常见的修正模式：

- "太细了" → 合并同类项，只保留关键结果
- "不是我要做的" → 移到背景说明
- "缺少标题" → 补上项目名和任务标题
- "层级不对" → `#` 项目 → `##` 任务 → 背景说明 → `- [ ]` 列表

## 约束

- 始终用中文与用户对话
- 不要修改日志中已有的内容，只追加
- 不要使用 frontmatter/YAML 头
- 不要使用 callouts（`> [!note]` 等）
- 如果日志里已有同一项目的待办，合并到该项目下
